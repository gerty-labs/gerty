### 1. Blue Matador — Node OOM (Fluentd)
Source: https://www.bluematador.com/blog/post-mortem-kubernetes-node-oom
Value: HIGH (resource numbers, OOM mechanics)

Timeline: June 15-16, 2019
- 5:12 PM: First SystemOOM event
- 6:02 PM next day: Second SystemOOM on different node

Resource Config (fix):
- Requests: 128Mi memory, 100m CPU
- Limits: 1024Mi memory, 250m CPU
- Node memory utilization: ~60-70% after recovery
- EBS Burst Balance lag: 10-15 minutes from CloudWatch

Root Cause: Fluentd-Sumologic pods had NO resource limits configured.
Fix: Added resource limits + env RUBY_GC_HEAP_OLDOBJECT_LIMIT_FACTOR=0.9
Key Lesson: K8s terminated memory-consuming pods while preserving production.

Training pairs: ~5
- No resource limits → SystemOOM (why limits matter)
- Fluentd/log shipper memory patterns
- Monitoring lag (CloudWatch 10-15min) masking resource exhaustion
- Right-sizing log shippers (128Mi/100m → 1024Mi/250m)
- EBS Burst Balance as leading indicator

### 3. deploy.live — Conntrack Exhaustion (GKE)
Source: https://deploy.live/blog/kubernetes-networking-problems-due-to-the-conntrack/
Value: HIGH (conntrack + resource interaction)

Scale:
- Baseline: 10,000+ RPS
- Peak: 4x baseline (40,000+ RPS)
- Downstream services: 100+ instances each

Conntrack Config:
- Default limit: 65k entries
- Formula: CONNTRACK_MAX = RAMSIZE(bytes) / 16384 / (x/32) where x=pointer bits
- GKE: ~5 conntrack_max entries per MB of RAM
- Target: 262,144 conntrack_max

Alert Threshold: >75% conntrack utilization

Root Cause: Node RAM halved + doubled downstream connections = conntrack exhaustion
Fix: Higher-RAM nodes + pod density limits; K8s 1.15+ upstream fix (PR #74840)
Monitoring: (node_nf_conntrack_entries / node_nf_conntrack_entries_limit) * 100

Training pairs: ~5
- Conntrack limit vs node RAM relationship
- 75% conntrack utilization alert threshold
- RAM reduction → conntrack reduction → packet loss
- Monitoring query for conntrack saturation
- K8s 1.15 conntrack fix

### 7. philpearl — Ingress Endpoint Race
Source: https://philpearl.github.io/post/k8s_ingress/
Value: MEDIUM (conceptual, no hard numbers)

Recommendation: Pods should ignore SIGTERM and continue serving until grace period expires.
Config: terminationGracePeriod must exceed load balancer reprogram time.

Source: https://oswalt.dev/2018/12/december-4-nre-labs-outage-post-mortem/
Value: HIGH (liveness probe misconfiguration)

Cluster: v1.12.1, single-master kubeadm, 3 masters + 5 workers (GCP)

etcd Liveness Probe:
- Command: etcdctl --endpoints=https://[127.0.0.1]:2379 get foo
- failureThreshold: 8
- timeoutSeconds: 15
- initialDelaySeconds: 15
- Critical calculation: 8 × 15s = 120-second restart cycle

Restart Metrics:
- etcd: 183 restarts
- API server: 183 restarts
- Controller-manager: 91 restarts
- Uptime per cycle: 1.5-2 minutes before SIGTERM

Worker version mismatch: one worker at v1.11.3 while others at v1.12.1

Fix: Removed entire liveness probe section from etcd manifest.

Training pairs: ~4
- Liveness probe failure threshold calculation (8 × 15s = 120s)
- etcd restart cascade (183 restarts)
- Liveness probe anti-patterns
- Master component restart cascade

### 9. tibobeijen — ElastAlert OOM (AWS)
Source: https://www.tibobeijen.nl/2019/02/01/learning-from-kubernetes-cluster-failure/
Value: HIGH (missing limits + cascade)

Cluster: v1.10.6, 8 nodes (3 masters + 5 workers), AWS eu-west-1
Masters: 43 days old; workers: 2-4 hours old

OOM Events: 4 SystemOOM events in 36-minute window
Affected: 2 nodes entered NotReady

Root Cause: ElastAlert pods — Helm chart supports resource limits but doesn't set defaults.
Data volume: "Millions of requests/hour" through CloudFront
Memory behavior: unbounded growth ("lines going up into nowhere")
Missing: use_count_query and max_query_size parameters

Cascade of Failures:
1. No resource limits on ElastAlert
2. No LimitRange for namespace
3. No CPU/memory/disk alerts
4. In-cluster Grafana/Prometheus unreachable during stress
5. Isolated deployment without team awareness
6. No post-deployment smoke tests

Fixes:
- Resource limits on ALL Helm deployments
- Namespace-level LimitRange defaults
- Monitoring infrastructure OUTSIDE cluster
- GitOps deployment approach
- Segregate ops tools to dedicated node group

Training pairs: ~6
- Helm chart without default resource limits → OOM
- ElastAlert memory unbounded growth pattern
- LimitRange as namespace safety net
- Monitoring inside cluster becomes unreachable during outage
- Ops workloads on dedicated node groups
- 4 SystemOOM events in 36 minutes cascade

### 10. Grafana — Pod Priority Cascade
Source: https://grafana.com/blog/2019/07/24/how-a-production-outage-was-caused-using-kubernetes-pod-priorities/
Value: HIGH (priority eviction cascade)

Resource Config:
- Ingester pod: 4 cores CPU + 15GB RAM per pod (25% of single machine)
- Cluster had unused aggregate resources but no 25% contiguous space

Priority Classes: Critical > High > Medium > Low
- Critical: monitoring (Prometheus, Alertmanager, node-exporter, kube-state-metrics)
- High: Ingesters
- Medium: default (unspecified pods, stateless microservices)

Cascade Mechanism:
1. New cluster's medium-priority Ingesters preempted production untagged Ingesters
2. ReplicaSet created replacement with default (medium) priority
3. Replacement preempted another production Ingester
4. Self-perpetuating cascade preempted ALL Ingester pods

Data: Ingesters hold up to 12h uncompressed data; no data lost (flushed in time)

Timeline:
- July 18: Priority classes deployed
- July 19: New cluster without priority designations triggered cascade
- ~4 min: Error alerts
- ~5 min: Diagnosis + scaling
- ~5 min: Ingesters flushed data
- ~10 min: Reverse proxy OOM resolved
- Total outage: 26 minutes

Training pairs: ~5
- Pod priority cascade mechanism
- Ingester: 4 CPU + 15GB = 25% of node (fragmentation risk)
- Default priority class danger
- Priority class deployment across clusters
- 26-minute recovery with proper flushing

### 15. Slack Jan 2021 — Transit Gateway Saturation
Source: https://slack.engineering/slacks-outage-on-january-4th-2021/
Value: HIGH (autoscaling cascade)

Normal message success: >99.999%
Outage success: 99% at 6:57am PST
Scale-up attempt: 1,200 servers between 7:01-7:15am

Cascade Mechanism:
1. AWS Transit Gateway saturated → packet loss
2. Higher latency → lower CPU → autoscaler DOWNSCALED (wrong direction!)
3. Then thread utilization spiked → aggressive upscale
4. 1,200 simultaneous provisions overwhelmed provision-service
5. Provision-service hit Linux open files limit + AWS quota limit
6. Unprovisionable instances exceeded ASG size limits
7. Monitoring dashboards in separate VPC, dependent on degraded TGW
8. Engineers lost SSH when instances deprovisioned

Recovery: Load balancer "panic mode" (balance across all instances when many failing health checks)

Training pairs: ~5
- Autoscaler misreading low CPU during saturation
- Provision-service resource limits (file descriptors)
- Monitoring in separate VPC dependency
- Transit Gateway as single point of failure
- Panic mode recovery pattern

### 16. Slack Feb 2022 — Consul + Cache Cascade
Source: https://slack.engineering/slacks-incident-on-2-22-22/
Value: MEDIUM (cache + service discovery interaction)

Root Cause: Consul agent 25% rollout → memcached node churn → cache miss → DB overload → timeout → can't refill cache → metastable failure

Infrastructure:
- Memcached fronted by Mcrouter (consistent hashing)
- Mcrib control plane generates Mcrouter configs from Consul catalog
- Vitess (horizontal MySQL) with user-sharded keyspace
- "Scatter query" hitting every shard for membership data

Cascade: Lower cache hit → higher DB load → timeouts → can't fill cache → loop
Client retries worsened the cascade.

Fix: Restrictive boot rate limits + query optimization (read replicas)

Training pairs: ~3
- Cache churn → metastable failure loop
- Service discovery (Consul) interaction with caching
- Scatter query anti-pattern

### 21. Spark on K8s (Azure) — OOM from tmpfs
Source: https://dev.to/pranavbhasker/postmortem-eliminating-oom-failures-in-spark-on-kubernetes-azure-after-cloud-migration-5fia
Value: MEDIUM (tmpfs + anti-affinity)

Node RAM: 64GB; Shuffle partitions: spark.sql.shuffle.partitions=200
Root Cause: emptyDir.medium: Memory (tmpfs) + missing pod anti-affinity → all executors on one node

Fix:
- Anti-affinity: preferredDuringSchedulingIgnoredDuringExecution weight:100, topologyKey: kubernetes.io/hostname
- Storage: emptyDir.medium: Memory → emptyDir: {} (disk-backed)

Result: 6 months stable, zero OOM incidents

Training pairs: ~3
- tmpfs emptyDir eats from container memory limit
- Pod anti-affinity for Spark executors
- emptyDir Memory vs disk-backed tradeoff
